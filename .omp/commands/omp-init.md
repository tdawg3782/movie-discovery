You are the PROJECT ONBOARDING pass for this repository. Run ONCE after scaffolding (or re-run anytime the stack changes). Your job: understand what this project IS, then wire the orchestrator's `.omp/` to it. This is READ-mostly - your ONLY writes are `.omp/project-profile.md` and replacing `__OMP_DETECT__` sentinels inside `.omp/`. Do NOT edit application code, run builds, or commit.

## 1. Survey (read-only)
Read what exists; prefer the project's own signals over assumptions. Fan out read-only `explore` agents for a large tree.
- Docs: `CHANGELOG.md`, `README.md`, `CLAUDE.md` / `AGENTS.md` (authoritative if present), `docs/`.
- Dependency manifests: `pyproject.toml`, `requirements*.txt`, `setup.cfg`, `package.json`, `*.csproj` / `*.sln`, `go.mod`, `Cargo.toml`, `Gemfile`, `pom.xml`, `build.gradle`, `composer.json`.
- Test setup: test directories, test config (`pytest.ini` / `tox.ini` / `[tool.pytest]`, `vitest`/`jest` config, `*.Tests.csproj`), and any `scripts.test` / Makefile / task-runner targets.
- Source layout: identify backend vs frontend and any monorepo package boundaries.

**Greenfield fallback (empty or near-empty repo):** if there is no code/manifest to read, the stack has still been DECIDED during planning - read `docs/superpowers/specs/*` and `docs/superpowers/plans/*` (newest first) plus anything the user stated, and derive the profile PROSPECTIVELY from that decision. Mark the profile `Source: spec/plan (greenfield)`.

## 2. Understand
Determine, citing the evidence you read:
- **Backend** stack (language + version + datastore) and **Frontend** stack (framework + build tool), or "none" for either.
- Where backend tests and frontend tests live.
- The **fast** command - the sub-minute filtered/unit run used as the per-task gate - and the **full** command - the complete suite (integration / e2e / golden) used at the barrier. Derive both from the project's OWN scripts/markers, not a hardcoded table. Shapes to adapt (do not copy blindly):
  - pytest: fast `cd <pkg> && pytest -q` (optionally `-m "not slow"`), full `cd <pkg> && pytest`.
  - node: fast `npm test` or `vitest run`, full adds e2e (`playwright test`) when present.
  - dotnet: fast `dotnet test --filter <excludes integration/golden>`, full `dotnet test`.
- Monorepo with BOTH a backend and frontend runner: the fast/full pair targets the PRIMARY area; record both runners in the profile.

If the test runner is genuinely ambiguous (two equal primaries) or NO tests exist, STOP and ASK rather than guessing.

## 3. Write the profile
Write `.omp/project-profile.md` (overwrite if present - this command is idempotent):

```
# Project Profile (auto-generated by /omp-init)
Backend:   <stack / version / datastore, or none>
Frontend:  <framework + build tool, or none>
Tests:     <backend test location> / <frontend test location, or none>
Fast:      <fast command>
Full:      <full command>
Conventions: <2-6 project-specific rules worth knowing every slice: naming, banned APIs, env layout, client base classes, etc.>
Source:    <code | spec/plan (greenfield)>
```
Keep it tight - it is a supplement the agents read every slice; CLAUDE.md remains authoritative.

## 4. Fill the sentinels
Replace `__OMP_DETECT__` placeholders inside `.omp/` with the detected commands, under two rules:
- **Skip this command file.** Do NOT edit `.omp/commands/omp-init.md` - its `__OMP_DETECT__` mentions are documentation, not placeholders. Fill every OTHER file under `.omp/`.
- **FAST vs FULL by context.** Use the FAST command wherever the per-task gate / scoped run is referenced, and the FULL command wherever the barrier / full suite is referenced. Some lines carry BOTH on one line (e.g. "...use `__OMP_DETECT__` scoped to the task; the barrier uses `__OMP_DETECT__`") - there the first (scoped / per-task) becomes FAST and the second (barrier / full) becomes FULL. Read the surrounding words; never blind-replace.
After filling, confirm NO `__OMP_DETECT__` remains anywhere under `.omp/` EXCEPT inside `.omp/commands/omp-init.md`. If there were no placeholders to begin with (the user passed `--test-fast` / `--test-full` to `init.py`), skip the fill - just record the existing commands in the profile.

## 5. Verify
Run the **fast** command and report PASS/FAIL with the actual output.
- Green -> done. Report the stack you found, the commands you wired, and the profile path.
- Red but the project HAS a suite -> the command shape is likely wrong; correct the profile + sentinels and re-run. Never leave a known-bad command in place.
- **Greenfield / no harness built yet** -> do NOT claim green. Set commands prospectively, write the profile, and report: "commands set from the plan; will validate after the first slice builds the test harness."

## Output
End with a short report: detected backend/frontend, fast/full commands wired, `.omp/project-profile.md` written, and the verify result (green / red / prospective). If you hit real ambiguity, surface the question instead of writing a guess.
