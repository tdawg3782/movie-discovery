# Code Audit — 2026-06-10 (run 2)

**Stack:** FastAPI (Python 3.11/3.12) + SQLAlchemy 2.x over SQLite / Vue 3 (`<script setup>`) + Vite. External APIs: TMDB, Radarr, Sonarr via `BaseArrClient`.

**Dimensions run (universal):** correctness & logic, concurrency/async, data integrity & persistence, error handling & resilience, security & privacy, type safety & contracts, performance hot paths, test coverage gaps, dead/unused code.

**Project-specific dimensions (derived from CLAUDE.md / project profile):**
1. **Settings propagation (env vs DB split-brain)** — driven by the dual config sources: `.env` in project root *and* UI-managed encrypted Settings rows. Every client construction was checked against both sources.
2. **External-API client lifecycle & response-shape assumptions** — driven by the `BaseArrClient` "persistent httpx client" convention and `_get_or_none` 404 contract; checked client reuse/closure and unguarded field access on TMDB/Radarr/Sonarr payloads.
3. **TMDB id-namespace integrity** — driven by the documented "show"→"tv" convention: movie and TV ids are separate TMDB namespaces, so any keying by bare `tmdb_id` is suspect.

Migration discipline check (the v2.4.1 landmine): `_migrate_watchlist_columns()` correctly covers `priority` and `tags`, runs after `create_all`, is idempotent, and has a dedicated test (`test_watchlist_migration.py`). No finding — verified clean.

No specialized domain auditor in `.omp/agents/` (only workflow agents: implementer / test-writer / reviewer).

**Fix first:** make all TMDB/Radarr/Sonarr client construction read `get_setting()` (DB-first) instead of env-only `settings.*` — UI-saved API keys and URLs are currently stored, pass the "Test" button, and are then silently ignored by the entire app.

## High

- [ ] **UI-configured API keys/URLs are never used by any client** — `backend/src/app/modules/discovery/router.py:12`, `calendar/router.py:18`, `recommendations/router.py:19`, `watchlist/router.py:107,126,170`, `radarr/router.py:12`, `sonarr/router.py:12`, `watchlist/service.py:116,120`. Every `TMDBClient`/`RadarrClient`/`SonarrClient` is built from env-only `app.config.settings.*`; the only consumers of DB-stored values are `/api/settings/test` (settings/router.py:53–79) and the root-folder/quality-profile/streaming-region lookups. **Trigger:** follow the documented First-Time Setup (README "enter your API keys via UI, click Test, Save") with no `.env` — Test succeeds, every feature then fails with empty/default credentials. With both sources present, a key rotated via UI silently keeps using the stale env value. The discovery/calendar/recommendations clients are also frozen at import time, so even env changes need a restart. **Impact:** documented configuration path is non-functional; "Test passed" actively misleads. **Fix:** resolve credentials per request via `get_setting("tmdb_api_key")` etc. (e.g. in the `get_radarr_client`/`get_sonarr_client` dependencies and a `get_tmdb_client` dependency replacing the module-level singletons), or load DB values into the shared clients on settings save. Add a regression test: save key via `SettingsService`, assert the client dependency picks it up.

## Medium

- [ ] **Watchlist keyed by bare `tmdb_id`, ignoring media_type namespace collision** — `backend/src/app/modules/watchlist/service.py:57-59` (`get_by_tmdb_id` filters only `tmdb_id`), used by `update_seasons` (:63), `process_batch` (:133, :152), and `delete_batch` (:101 — `tmdb_id.in_(ids)` with no `media_type` filter). TMDB movie and TV ids are separate dense namespaces (movie 603 and tv 603 are different titles). **Trigger:** a movie and a show with the same numeric id both on the watchlist — batch-deleting one deletes both rows; `PATCH /{tmdb_id}/seasons` and the post-process `status="added"` write may hit the wrong row. Frontend selection has the same keying (`WatchlistView.vue:431-448` `isSelected`/`toggleSelectAll` by `tmdb_id` only). **Impact:** silent cross-type data loss / wrong-row updates. **Fix:** add `media_type` to `get_by_tmdb_id`, `delete_batch` (request schema already implies per-type batches; pass it through), `update_seasons` route, and the frontend selection key; add a unique constraint on `(tmdb_id, media_type)` via the additive-migration path.

- [ ] **`update_season_monitoring` crashes on `None` seasons; surfaces as cryptic per-item failure** — `backend/src/app/modules/sonarr/client.py:187` does `season.get("seasonNumber") in seasons_to_add` where `seasons_to_add` can be `None`: `watchlist/service.py:134-138` passes `selected_seasons=None` when the row's JSON is NULL (reachable: `PATCH /{tmdb_id}/seasons` accepts `selected_seasons: null` and stores NULL on an `is_season_update` item). **Trigger:** process such an item → `TypeError: argument of type 'NoneType' is not iterable`, caught by `process_one`'s blanket except and reported as `{"error": "argument of type 'NoneType' is not iterable"}`. **Impact:** user-facing gibberish error; the intended "all seasons" semantics never happen. **Fix:** in `process_batch`, treat `None` as "monitor all" (or reject season-updates without seasons with a clear message); guard `seasons_to_add or []` in the client.

- [ ] **Calendar includes already-processed watchlist movies (status filter checks a value that is never set)** — `backend/src/app/modules/calendar/router.py:41` filters `item.status != "available"`, but watchlist statuses are only `pending`/`added`/`downloading` (`models.py:52`; `process_batch` sets `"added"`). **Trigger:** process a watchlist movie to Radarr, open Coming Soon — the movie appears twice: once from the Radarr calendar (`in_library: True`) and once as a watchlist entry (`in_library: False`), contradicting README ("watchlist movies *not yet in your library*"). **Impact:** duplicate agenda entries with conflicting library status. **Fix:** filter `item.status == "pending"` (or `not in ("added", "downloading")`).

- [ ] **Calendar 500s if a Sonarr record lacks season/episode numbers** — `backend/src/app/modules/calendar/service.py:41`: `f"S{season_number:02d}E{episode_number:02d}"` raises `TypeError` when `record.get("seasonNumber")` or `episodeNumber` is `None`. Every other field in the normalizer is null-guarded; these two aren't, and one bad record kills the entire `/api/calendar` response (no per-record try). **Trigger:** any Sonarr calendar record missing either field (specials/malformed entries). **Impact:** whole Coming Soon page errors. **Fix:** guard — `subtitle = f"S{sn:02d}E{en:02d} · {title}" if sn is not None and en is not None else episode_title`.

- [ ] **Per-request httpx.AsyncClient instances are never closed** — `radarr/router.py:12` and `sonarr/router.py:12` construct a new client per request via `Depends` (so the `BaseArrClient` "persistent connection reuse" only lasts one request); `watchlist/router.py:107,126,170` and `watchlist/service.py:116,120` create `TMDBClient`/arr clients per call. None call `.close()`; pools are released only by GC. Meanwhile discovery/calendar/recommendations correctly share module-level TMDB clients. **Trigger:** normal traffic; each watchlist page view leaks one TMDB client + its pool. **Impact:** "Unclosed AsyncClient" warnings, socket churn, defeats the documented connection-reuse pattern. **Fix:** share module-level (or app-state) clients for all three services, or make the dependencies async generators that `await client.close()` after the response. (Pairs naturally with the High fix.)

- [ ] **Stale-response race in DiscoverView fetches** — `frontend/src/views/DiscoverView.vue:220-255` (`fetchContent`) and `:179-202` (`performSearch`) have no request token/abort; `switchTab`/`goToPage`/`handleFilterChange` fire overlapping requests. **Trigger:** switch tabs (or pages) quickly on a slow connection — the first, slower response resolves last and overwrites `items`/`totalPages` for the wrong tab/page while the URL says otherwise. **Impact:** view shows wrong tab's content; pagination desyncs. **Fix:** request-sequence token (`const seq = ++latest; ... if (seq !== latest) return`) or AbortController per fetch.

- [ ] **DiscoverView ignores browser Back/Forward within the view** — `frontend/src/views/DiscoverView.vue:88,339-342`: `route.query` is parsed once on mount; every page/tab/filter change does `router.push({ query })` (new history entries) but there is no `watch(() => route.query, …)`. **Trigger:** page 1 → 2 → 3, press Back — URL shows `page=2`, view stays on page 3. (Back from a detail page works only because the component remounts.) README explicitly promises Back-button restoration. **Impact:** documented navigation contract broken inside the view. **Fix:** watch `route.query`, re-parse with `parseDiscoverState`, and refetch when it differs from current state (guard against self-induced pushes).

- [ ] **For You silently drops the owned-library exclusion when Radarr/Sonarr are unreachable** — `backend/src/app/modules/recommendations/router.py:42-53`: both `get_all_movies()`/`get_all_series()` calls are wrapped in `except Exception: pass`. **Trigger:** Radarr down or misconfigured → `owned_keys` empty → recommendations include titles the user already owns, violating the documented "titles already owned … are never shown" contract — and the degraded result is then cached for 6h. **Impact:** silent contract violation, no surfaced warning. **Fix:** log the failure and either surface a `degraded: true` flag in the response or skip caching when an exclusion source failed.

- [ ] **`GET /api/watchlist` makes one TMDB call per item on every request; `MediaCache` table exists but is unused** — `backend/src/app/modules/watchlist/router.py:107-108` enriches every row with a live `get_details` call (parallel, but uncached). **Trigger:** 100-item watchlist → 100 TMDB requests per page view (and per details-save, which re-fetches the whole list); TMDB rate limit is ~40 req/10s, so large lists intermittently degrade to the `TMDB:{id}` fallback title via the blanket except in `_enrich_watchlist_item`. **Impact:** slow watchlist loads, rate-limit-induced metadata loss. **Fix:** use the already-defined `MediaCache` model (models.py:24) as a read-through cache keyed by `(tmdb_id, media_type)` with `cached_at` TTL.

- [ ] **`get_setting` leaks the DB session on error and swallows all failures** — `backend/src/app/config.py:46-53`: `db.close()` is not in a `finally`; any exception (e.g. Fernet `InvalidToken` after the `data/.encryption_key` file is lost/regenerated) leaks the session and silently falls back to the env value. **Trigger:** encryption key file replaced (fresh volume mount, restored DB without `data/.encryption_key`) → decryption fails on every call, masked by env fallback or `None`, with zero log output. **Impact:** undiagnosable "settings reverted" behavior; session leak per failing call. **Fix:** `try/finally` around the session; catch only "table missing" cases narrowly and log decryption failures.

## Low

- [ ] **Duplicate-add race on watchlist** — `backend/src/app/modules/watchlist/service.py:27-47`: check-then-insert with no unique constraint on `(tmdb_id, media_type)`. Two concurrent adds (double-click) create duplicate rows. **Fix:** unique index via the additive-migration path + `IntegrityError` catch returning the existing row.

- [ ] **Dead models: `MediaCache` and `LibraryStatus`** — `backend/src/app/models.py:24,65`: defined and table-created, referenced nowhere in `backend/src`. Either wire `MediaCache` into watchlist enrichment (see Medium above) or delete both models.

- [ ] **`_parse_seasons` accepts non-list JSON** — `backend/src/app/modules/watchlist/router.py:33-40`: `json.loads("5")` returns `5` (int) which is then passed around as `selected_seasons`. Validate `isinstance(result, list)` like `_parse_tags` does.

- [ ] **Search response pagination metadata is inconsistent** — `backend/src/app/modules/discovery/router.py:152-160`: person results are filtered out but `total_pages` comes from TMDB while `total_results=len(results)` counts only the current filtered page. Harmless today (frontend uses `total_pages` only), but the two fields disagree by construction.

- [ ] **Settings update silently ignores attempts to clear non-clearable keys** — `backend/src/app/modules/settings/service.py:53-61`: clearing e.g. `radarr_url` (empty string) is a silent no-op; the UI gets back the old value with no error. Return/log which fields were skipped.

## Test coverage gaps

- [ ] **No test asserts DB-saved settings reach the API clients** — exactly the High finding's blind spot: `test_settings_service.py` proves persistence, `test_settings_router.py` proves the test endpoint, nothing proves consumption. Add an integration test: save a key via `SettingsService`, assert the client dependency/factory uses it.
- [ ] **`process_batch` season-update edge untested** — no test covers `is_season_update=True` with NULL `selected_seasons` (the Medium TypeError path) nor the failure shape returned to the UI.
- [ ] **Calendar normalizers lack malformed-record cases** — `test_calendar_service.py` covers happy paths; no record-missing-`seasonNumber`/`episodeNumber` case (the Medium 500).
- [ ] **`delete_batch`/`update_seasons` cross-media-type scoping untested** — no test pins that a movie and show sharing a `tmdb_id` are independent (the keying Medium).
- [ ] **Frontend views untested** — Vitest covers only `utils/*` (good, pure). The race-prone view logic (DiscoverView fetch sequencing, back-button restore) has no coverage; the two Medium frontend bugs would need component tests or extraction of fetch-sequencing into a tested util.
