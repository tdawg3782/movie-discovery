# Code Audit — 2026-06-10

**Stack:** FastAPI / Python ≥3.11 / SQLAlchemy 2.x over SQLite (NAS Docker, Cloudflare tunnel) + Vue 3 / Vite.
**Dimensions run (universal):** correctness & logic, concurrency/async, data integrity & persistence, error handling & resilience, security & privacy, type safety & contracts, performance hot paths, test coverage gaps, dead code.
**Project-specific dimensions:** PS-1 Arr/TMDB client layer (BaseArrClient, `_get_or_none`, "show"→"tv"); PS-2 SQLite additive-migration discipline (v2.4.1 incident contract); PS-3 frontend state/fetch races & error surfacing.
**Method:** 8 parallel read-only subagents; every Critical/High re-verified against source by the auditor. Notable false positives dropped during verification: "lazy httpx client race" (no await between check and assign — atomic in asyncio), "missing timeout = infinite hang" in settings tests (httpx defaults to 5s), "TMDB key in URL" (TMDB v3's own auth convention), "encryption key committed to repo" (`data/` is gitignored).

> **Fix first:** Put an auth gate in front of the API (or verify Cloudflare Access already does), then fix H1 — UI-saved settings are silently never used by any API client.

---

## Critical

- [ ] **C1. No authentication on any endpoint, canonically deployed behind a public Cloudflare tunnel** — `backend/src/app/main.py:28-52` registers every router with zero auth middleware; nginx (`frontend/nginx.conf`) proxies `/api` straight through; CLAUDE.md/README state the canonical install is internet-reachable via Cloudflare tunnel. An anonymous visitor can: overwrite all settings (`PUT /api/settings`), drive Radarr/Sonarr (add movies/series, change season monitoring), wipe the watchlist (`DELETE /api/watchlist/batch`), and use the settings test endpoint as an internal-network probe (see H3). *Caveat: severity assumes no Cloudflare Access policy in front of the tunnel — this could not be verified from the repo. If Access is configured, downgrade to High (defense-in-depth).* **Fix:** add a simple bearer-token/API-key middleware on `app` (or document + enforce a Cloudflare Access policy as a deployment prerequisite in CLAUDE.md).

## High

- [ ] **H1. Settings configured via UI never reach any API client — documented feature is silently broken** — `SettingsService.update_settings` (`backend/src/app/modules/settings/service.py:49-60`) writes only to the DB and never touches the env-backed `settings` object, but every real client binds env values: arr factories `backend/src/app/modules/radarr/router.py:11-13` (+ sonarr equivalent) use `settings.radarr_url/_api_key`; `process_batch` (`backend/src/app/modules/watchlist/service.py:117-124`) likewise (while reading root_folder/quality_profile via DB-aware `get_setting` two lines later — inconsistent in the same function); module-level `TMDBClient(api_key=settings.tmdb_api_key)` is bound **at import** in `discovery/router.py:12`, `calendar/router.py:19`, `recommendations/router.py:20`, and per-request in `watchlist/router.py:107,126`. Only `POST /api/settings/test` uses `get_setting`, so the UI shows "connection successful" while every actual endpoint keeps using stale/empty `.env` values. README First-Time Setup (steps 2-6) and CLAUDE.md:117 document UI configuration as the primary path. **Fix:** route all client construction through `get_setting` (factories and `process_batch`), and replace import-time TMDB singletons with a per-request dependency (or a provider that re-reads the key on change).
- [ ] **H2. Migration discipline violation: `selected_seasons` and `is_season_update` have no guarded ALTER** — `backend/src/app/models.py:52-54` define both columns; `_migrate_watchlist_columns` (`backend/src/app/database.py:30-44`) guards only `priority` and `tags`. Exact v2.4.1 incident pattern: any watchlist DB created before these columns (e.g. restored backup) makes every watchlist read/write 500 with "no such column". The live NAS DB evidently has them (app works), so exposure is restores/older copies — but the CLAUDE.md contract says every such column MUST be guarded. **Fix:** add `selected_seasons TEXT` and `is_season_update BOOLEAN NOT NULL DEFAULT 0` guards to `_migrate_watchlist_columns()` and cover them in `test_watchlist_migration.py`.
- [ ] **H3. SSRF / internal-network probe via settings** — `PUT /api/settings` accepts arbitrary `radarr_url`/`sonarr_url` (no URL validation in `SettingsUpdate`), then `POST /api/settings/test` (`backend/src/app/modules/settings/router.py:34-46`) GETs `{url}/api/v3/system/status` and discloses the outcome — status code on response, raw `str(e)` on failure (`router.py:46,64`) — a clean oracle for scanning the NAS LAN from the internet (amplified by C1). **Fix:** validate URLs with Pydantic `HttpUrl` + scheme/host allowlist (RFC1918-only or explicit allowlist), and return generic failure messages (log details server-side).
- [ ] **H4. One unreachable *arr service 500s entire combined endpoints; queue/recent/seasons endpoints have no error handling at all** — `calendar/router.py:33-36` and `library/router.py:22-26,40-43` use `asyncio.gather` without `return_exceptions` or try/except: if either service is down (or simply unconfigured — factory defaults are `http://localhost:7878/8989`, guaranteeing `ConnectError`), Calendar and Library pages are fully dead even though the other source is healthy. `radarr/router.py:88-99` (queue, recent) and `sonarr/router.py:75-84,101-113` (seasons, queue, recent) propagate raw httpx errors as 500s. **Fix:** `gather(..., return_exceptions=True)` with per-source fallback (empty list + `partial_failure` flag) in calendar/library; wrap the bare endpoints in the same Timeout/HTTPStatusError/ConnectError → 503/504 handling already used in `add_movie`.
- [ ] **H5. Stale-response races in DiscoverView** — `frontend/src/views/DiscoverView.vue:179-202` (`performSearch`) and `:220-255` (`fetchContent`); `goToPage` (`:257-263`) fires either without cancelling in-flight requests. Rapid search/pagination/filter changes let a slow older response overwrite newer results — UI shows results that don't match the visible query/page. **Fix:** module-scoped monotonically increasing request id; discard responses whose id ≠ latest (or AbortController per fetch).

## Medium

- [ ] **M1. Encryption-key lifecycle: key loss permanently 500s the Settings page** — `encryption.py:7-21` binds a module-level `Fernet` at import from `data/.encryption_key`; `decrypt_value` (`:28-30`) propagates `InvalidToken`. `get_raw_value`/`_get_masked` (`settings/service.py:69-75,79-86`) don't catch it, so after a key regen/restore mismatch, `GET /api/settings` 500s with no recovery path — while `get_setting` (`config.py:50-56`) swallows the same failure and silently falls back to `.env` everywhere else. **Fix:** catch `InvalidToken` in `get_raw_value`/`_get_masked`, surface a "re-enter API keys" state, and delete the unrecoverable rows.
- [ ] **M2. Sonarr calendar entry with missing season/episode number crashes the agenda** — `calendar/service.py:38-41`: `record.get("seasonNumber")` can be `None`; `f"S{None:02d}..."` raises `TypeError` → whole `/api/calendar` 500s on one malformed record (specials/TBA episodes). **Fix:** default to 0 (`record.get("seasonNumber") or 0`) or skip subtitle when either is missing.
- [ ] **M3. `get_by_tmdb_id` ignores `media_type` — movie/TV id collision** — `watchlist/service.py:58-60` filters on `tmdb_id` only; TMDB movie and TV id spaces overlap. With both a movie and a show of the same id in the watchlist, `process_batch` (`:143,151,167`) and `update_seasons` (`:62-70`, exposed as `PATCH /api/watchlist/{tmdb_id}/seasons`) can read/update the wrong row (wrong seasons sent to Sonarr, wrong item marked "added"). **Fix:** add a `media_type` parameter and filter on both columns; index `(tmdb_id, media_type)`.
- [ ] **M4. Calendar watchlist enrichment: serial N+1 TMDB calls, failures silently dropped** — `calendar/router.py:40-52` awaits `get_details` per watchlist movie in a loop (no gather) and `except Exception: continue` hides every failure — TMDB down ⇒ watchlist releases silently vanish from the agenda (indistinguishable from "none upcoming"). **Fix:** `asyncio.gather(..., return_exceptions=True)` over the filtered items; count failures and surface a partial-failure flag.
- [ ] **M5. MediaDetailView doesn't refetch on cross-type navigation with same id** — `/movie/:id` and `/tv/:id` share the component (`router/index.js`), and the watcher (`MediaDetailView.vue:174-177`) observes only `route.params.id`. Navigating `/tv/123` → `/movie/123` (possible via recommendation/cast links) keeps the old show's data under the new URL. **Fix:** `watch([() => route.params.id, () => route.meta.mediaType], ...)`.
- [ ] **M6. `get_setting` swallows every exception** — `config.py:50-56` `except Exception: pass` hides DB corruption, decrypt failures, and schema errors, silently degrading to `.env`. Combined with H1 this made the settings-propagation bug invisible. **Fix:** log the exception (once per key) before falling back.
- [ ] **M7. Test gaps where the audit found the real bugs** — zero tests for `library/router.py` (both endpoints 500 on one arr down — H4 would have been caught); settings `test_connection` success paths untested; encryption failure paths (missing/corrupt key, `InvalidToken`) untested; arr/TMDB client tests patch the client's own `_get/_post` so response-shape parsing and error mapping are never exercised (tautological); frontend: zero tests for all services, views, and components (only 4 state utils covered). **Fix:** prioritize library-router tests with one-service-down fixtures, encryption failure tests, and a thin httpx-mock (respx) layer for one client as the pattern.
- [ ] **M8. For You: cache hit still pays full seed computation** — `recommendations/router.py:38-55` fetches all Radarr movies + all Sonarr series + watchlist on every request before the cache check at `:62-64`; the 6h TTL cache only saves the TMDB fan-out. Concurrent cold-cache requests also duplicate the whole TMDB fan-out (no in-flight dedup; dict writes themselves are atomic — correctness is fine). **Fix:** cache the seed/exclusion computation alongside (short TTL), or accept and document; optionally guard recompute with an `asyncio.Lock`.

## Low

- [ ] **L1. `POST /api/watchlist` returns 201 for duplicates** — `watchlist/router.py:113` fixes `status_code=201`; `service.add` (`service.py:27-33`) returns the existing row. **Fix:** return 200 for existing (or 409), e.g. set response status conditionally.
- [ ] **L2. Search `total_results` is the filtered page count** — `discovery/router.py:157` reports `len(results)` (current page, person-filtered) as `total_results` instead of TMDB's total. **Fix:** pass through `data["total_results"]` or rename the field.
- [ ] **L3. Dead code: `MediaCache` model and `appStore.js`** — `models.py:24-38` (zero references outside the definition) creates an unused table; `frontend/src/stores/appStore.js` exports `useAppStore` that nothing imports (pinia is a dependency only for this). **Fix:** delete both (and drop pinia if nothing else uses it).
- [ ] **L4. `normalize_radarr` has no end-bound on the chosen date** — `calendar/service.py:69` keeps the earliest candidate `>= start` only; a record whose only future date is past `end` can leak into the agenda (rare — Radarr pre-filters the window). **Fix:** `start <= d <= end`.
- [ ] **L5. Connection-test responses leak raw exception text** — `settings/router.py:46,64` return `str(e)`. Mostly the user's own URLs, but it is also the oracle detail for H3. **Fix:** generic messages; log specifics server-side.
- [ ] **L6. Silent client-side/data degradation** — `_parse_seasons`/`_parse_tags` (`watchlist/router.py:38-47,54-61`) return defaults on corrupt JSON with no log; `fetchLibraryStatuses` (`DiscoverView.vue:121-156`) catches and only `console.warn`s, so missing library badges look like "not in library". **Fix:** log with item id backend-side; surface a non-blocking toast/flag frontend-side.
- [ ] **L7. Hardening gaps** — `frontend/nginx.conf` sets no security headers (X-Frame-Options, X-Content-Type-Options, CSP); protected arr endpoints (`add_movie`, batch status, quality-profiles) handle Timeout/HTTPStatusError but not `ConnectError` → 500 instead of 503 when the service is down. **Fix:** add the standard header set in nginx; add `except ConnectError → 503` to the existing handlers.
