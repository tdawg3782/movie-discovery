# Code Audit — 2026-06-10 (run 3)

**Stack:** FastAPI (Python 3.11 Docker / 3.12 local) + SQLAlchemy 2.x over SQLite (`data/` DB on the NAS, canonical deploy is internet-reachable via Cloudflare tunnel → nginx → FastAPI) / Vue 3 (`<script setup>`) + Vite + Axios. External APIs: TMDB, Radarr, Sonarr via `BaseArrClient`.

**Dimensions run (universal):** correctness & logic, concurrency/async/races, data integrity & persistence, error handling & resilience, security & privacy, type safety & contracts, performance hot paths, test coverage gaps, dead/unused code & state.

**Project-specific dimensions (derived from CLAUDE.md / project profile):**
1. **External-API client layer (PS-1)** — driven by the `BaseArrClient` "persistent httpx client" convention, the `TMDBClient._get_or_none()` 404 contract, and the "show"→"tv" boundary. Audited client lifecycle/closure, error/timeout handling, and response-shape assumptions.
2. **Settings env/DB split-brain + migration discipline (PS-2)** — driven by the dual config source (`.env` root file *and* UI-managed encrypted Settings rows; `get_setting()` is DB-first w/ env fallback) and the documented additive-migration contract (the v2.4.1 "no such column" incident).
3. **Frontend state/data-fetch races & error surfacing (PS-3)** — driven by the URL-as-state promise (reload + browser Back must restore state) and on-demand fetching across views.

**Method:** 12 parallel read-only `auditor` subagents (one per dimension), each pinned to a strong model. The **Security** auditor refused twice (its threat-model phrasing tripped a model safety filter), so that dimension was audited directly on this session's model against `main.py`, `settings/router.py`, `settings/schemas.py`, `encryption.py`, `nginx.conf`, `.gitignore`, `docker-compose.yml`. Every Critical/High below was re-read against source by the orchestrator before inclusion.

**False positives dropped during verification:** encryption key "committed to repo" (`.gitignore:12,32` ignores both `.env` and `data/`); TMDB `api_key` in query string (TMDB API v3's own auth scheme); "shared SQLAlchemy `Session` corrupted across `asyncio.gather` in `process_batch`" — the per-coroutine DB reads are synchronous and complete atomically between `await` points, and status writes run sequentially after the gather, so it is fragile but not a live bug on the single-worker deploy; raw-string SQL injection (all DB access is through the ORM).

**Cross-run note (important):** the same `selected_seasons`/`is_season_update` migration gap that **audit-2 declared "verified clean"** is confirmed real here (H3) — audit-2 only checked that `priority`/`tags` *were present*, not that the two earlier columns *also need guards*. `CHANGELOG.md:159-162` proves `is_season_update` was the original v2.4.1 incident. Audit-1's H2 was correct.

---

> **Fix first:** Put (or verify) an authentication layer in front of the public API — every endpoint is currently unauthenticated on an internet-reachable tunnel (C1). Then fix the settings split-brain (H1) so the documented UI setup actually works, and the bare-`tmdb_id` batch delete (H2) that silently destroys unrelated rows.

## Critical

- [ ] **C1. No authentication on any endpoint, on an internet-reachable deployment** — `backend/src/app/main.py:35-53` installs only `CORSMiddleware` (and CORS is *not* an auth control — non-browser clients ignore it entirely) and registers every router with no auth dependency/middleware; `frontend/nginx.conf:12-19` proxies `/api/` straight through. An unauthenticated visitor can overwrite all config (`PUT /api/settings`), wipe the watchlist (`DELETE /api/watchlist/batch`), drive Radarr/Sonarr (add movies/series, change season monitoring via the watchlist process flow), and use `/api/settings/test` as an internal-network probe (see M11). **Caveat:** severity assumes no Cloudflare Access policy fronts the tunnel — this cannot be verified from the repo. If Access is configured, downgrade to High. **Fix:** add an auth dependency (shared API key or session) applied to all routers (or a global dependency), or confirm and document a Cloudflare Access policy as the enforced gate.

## High

- [ ] **H1. UI-saved settings never reach any API client (env/DB split-brain)** — `backend/src/app/modules/discovery/router.py:12`, `calendar/router.py:18`, `recommendations/router.py:19` (module-level `TMDBClient(api_key=settings.tmdb_api_key)` bound **at import**), `radarr/router.py:10-12`, `sonarr/router.py:10-12` (per-request `Depends` factories), `watchlist/router.py:107,126,170`, and `watchlist/service.py:116,120` all build clients from the env-only `app.config.settings` object. Only `/api/settings/test` (`settings/router.py:53,68-79`) and `streaming_region`/`root_folder`/`quality_profile` read DB-first via `get_setting()`. **Smoking gun:** `process_batch` (`watchlist/service.py:116/120`) constructs the client from `settings.radarr_url`/`settings.radarr_api_key` but reads `root_folder`/`quality_profile_id` via `get_setting()` two lines down (`:117-118,121-122`) — split in one function. **Trigger:** follow README "First-Time Setup" (enter keys in UI, Test, Save, no `.env`) → Test goes green, then every feature 401s on TMDB or hits the `localhost:7878/8989` defaults with empty keys. With both sources present, stale env silently wins everywhere except Test (inverting the documented DB-first precedence); the import-time singletons additionally never pick up DB changes without a restart. **Fix:** resolve credentials at request time via `get_setting()` at each construction site — make the arr `Depends` factories DB-aware, add a `get_tmdb_client` dependency replacing the three module-level singletons, and fix the watchlist router + `process_batch`. *(Corroborates audit-1 H1 / audit-2 H1 — still unfixed.)*

- [ ] **H2. Watchlist batch-delete (and all `tmdb_id` lookups) ignore `media_type` → silent cross-namespace data loss** — `backend/src/app/modules/watchlist/service.py:97-105` (`delete_batch` filters `Watchlist.tmdb_id.in_(tmdb_ids)` with **no** `media_type`), `schemas.py:23` (`BatchDeleteRequest.ids` carries no media type), `:57-59` (`get_by_tmdb_id` filters only `tmdb_id`, used by `update_seasons` `:63` and `process_batch` `:133,:152`); frontend `WatchlistView.vue:416,431-433,436,447-451,508-509`. `add()` (`service.py:29`) dedupes on `(tmdb_id, media_type)`, so a movie and a show with the same numeric id **legitimately coexist** — TMDB movie/TV ids are separate dense namespaces (e.g. movie 603 ≠ tv 603). **Trigger:** have a movie and a show sharing a numeric id on the watchlist; batch-delete one. **Impact:** the other row is also deleted — permanent, unprompted loss of an unrelated item. Selection toggling collides (selecting the show toggles the movie's selection entry off); `PATCH /{tmdb_id}/seasons` and the post-process `status="added"` write can hit the wrong-type row. **Fix:** thread `(tmdb_id, media_type)` through `delete_batch` (have `BatchDeleteRequest` carry pairs, or send DB row ids — `removeSingle` already uses ids), `get_by_tmdb_id`, and `update_seasons`; key frontend selection by `${media_type}:${tmdb_id}`. *(Upgrades audit-1 M3 / audit-2 Medium → High: the batch-delete data-loss vector is the new evidence — four independent auditors converged on it.)*

- [ ] **H3. Missing additive-migration guards for `selected_seasons` and `is_season_update` (the v2.4.1 landmine, recurring)** — `backend/src/app/database.py:37-40` guards only `priority`/`tags`, but `models.py:51,53` also add `selected_seasons` and `is_season_update`, which post-date the original schema. `create_all` never adds columns to an existing table. **`CHANGELOG.md:159-162` confirms `is_season_update` *was* the v2.4.1 "500 on watchlist page load" incident** — it was patched manually but the fix was never folded into `_migrate_watchlist_columns()`. **Trigger:** restore/redeploy a pre-2.4.1 DB (old backup, fresh volume with an older copy) → every `Watchlist` SELECT raises `OperationalError: no such column: watchlist.selected_seasons` → `GET /api/watchlist` and batch processing 500. The live NAS DB currently has the columns (app works), so exposure is restores/older copies. **Fix:** add guarded ALTERs — `selected_seasons TEXT` and `is_season_update BOOLEAN NOT NULL DEFAULT 0`; better, derive missing columns generically from `Watchlist.__table__.columns` minus the inspector's columns so future columns can't be forgotten. *(Corroborates audit-1 H2; contradicts audit-2's "migration verified clean".)*

- [ ] **H4. `add()` silently discards new seasons on re-add → season-update data loss behind a 201** — `backend/src/app/modules/watchlist/service.py:32-33` (`if existing: return existing`) returns the existing row untouched, dropping the incoming `selected_seasons`/`is_season_update`. The documented season-update flow re-POSTs through this path (`frontend/src/views/MediaDetailView.vue` `handleAddWithSeasons`), but `process_batch` leaves processed rows in the list with `status="added"`, so a later "+S4" submit gets HTTP 201 with the stale row and the new selection is never persisted; the next process run calls `add_series` (flag still `False`) → "Series already in Sonarr library" instead of `update_season_monitoring`. **Fix:** when a duplicate exists and seasons/flag are provided, merge (union seasons, set `is_season_update`, reset `status="pending"`) — or return 409 so the frontend routes to the seasons PATCH. *(New — not in prior audits.)*

- [ ] **H5. One unreachable arr 500s the entire combined endpoint (no `return_exceptions`)** — `backend/src/app/modules/calendar/router.py:33-36` and `library/router.py:22-25,40-43` `asyncio.gather` the Radarr+Sonarr calls bare; `BaseArrClient._get` calls `raise_for_status` and lets `httpx.ConnectError` propagate, so if either service is down the exception escapes and the whole `/api/calendar`, `/api/library/activity`, `/api/library/queue` response 500s — discarding the healthy source and (for calendar) the arr-independent watchlist-movie agenda. **Fix:** `gather(..., return_exceptions=True)`, substitute `[]`/`{"records": []}` per failed source, and flag the degraded source so the frontend can show a banner instead of a fake-empty state. *(Corroborates audit-1 H4.)*

- [ ] **H6. radarr/sonarr endpoints don't handle `ConnectError`; queue/recent/seasons unguarded → raw 500** — `backend/src/app/modules/radarr/router.py:49-56,67-72,81-86` (and the sonarr mirror) catch only `TimeoutException`/`HTTPStatusError`; a down service raises `httpx.ConnectError` (connection refused/DNS — the most common failure), which falls through to a raw 500. `radarr/router.py:89-101` (`/queue`, `/recent`) and `sonarr/router.py:101-113` plus `/series/{tmdb_id}/seasons` (`:75-84`) have **no** try/except at all. The default factory URLs `localhost:7878/8989` guarantee `ConnectError` when unconfigured. **Fix:** catch `httpx.RequestError` → `HTTPException(503, "<svc> unreachable")` in the existing handlers and wrap the unguarded endpoints (ideally a shared app-level exception handler). *(Corroborates audit-1 H4.)*

- [ ] **H7. For You caches degraded/empty results for 6h and silently drops the owned-library exclusion** — `backend/src/app/modules/recommendations/router.py:41-54` wraps both `get_all_movies()`/`get_all_series()` in `except Exception: pass`, so an arr blip empties `owned_keys` → recommendations include already-owned titles (violating the documented "never shown" contract). Separately, `:70-86`: if all TMDB recommendation calls fail (`gather(return_exceptions=True)` then skip), an **empty** `MediaList` is cached under the seed signature for `RECS_CACHE_TTL = 6h` (`:84-86`) — a transient TMDB outage blanks For You for 6h after recovery; only `?refresh=true` clears it. **Fix:** only write `_cache` when `rec_results` is non-empty; log (and surface a `degraded` flag / skip caching) when an arr fetch fails instead of silently degrading. *(Corroborates audit-2 M8.)*

- [ ] **H8. `GET /api/watchlist` makes N live TMDB calls per request (unbounded); `MediaCache` table is unused** — `backend/src/app/modules/watchlist/router.py:100-110` enriches every row with a live `get_details` call via an **unbounded** `asyncio.gather`; `WatchlistView.vue` re-runs `fetchWatchlist()` after every details-save, seasons-save, process, and batch-delete, re-paying the full fan-out each time. A 50-item watchlist bursts 50 concurrent requests against TMDB's ~40 req/10s limit → 429s → rows degrade to the `TMDB:{id}` placeholder via the blanket except in `_enrich_watchlist_item`. The `MediaCache` model (`models.py:24`) was built for exactly this and has zero consumers. **Fix:** read-through `MediaCache` keyed `(tmdb_id, media_type)` with a TTL (title/poster/overview are near-static); bound residual fan-out with `asyncio.Semaphore(~8)`. *(Corroborates audit-2.)*

- [ ] **H9. Encryption key silently regenerated on loss → stored API keys permanently unreadable** — `backend/src/app/modules/settings/encryption.py:13-17` (`_get_or_create_key` transparently mints a new Fernet key whenever `data/.encryption_key` is absent — a hidden dotfile beside the DB, easy to miss in backups); `_fernet` is bound at import (`:20`) and `decrypt_value` (`:30`) raises `InvalidToken` uncaught. After a key regen/restore mismatch, `get_raw_value`/`_get_masked` (`settings/service.py:67-83`) propagate `InvalidToken` → `GET /api/settings` 500s with no recovery, while `get_setting()` (`config.py:45-57`) swallows the same error and silently falls back to `.env` everywhere else; re-saving re-encrypts under the new key, masking the destruction. **Fix:** on key creation, detect existing `encrypted=True` rows and fail loudly (or flag for re-entry); catch `InvalidToken` in the masked/raw getters and surface a "re-enter API keys" state; document `.encryption_key` as part of the backup set. *(Corroborates audit-1 M1 / audit-2.)*

- [ ] **H10. DiscoverView ignores browser Back/Forward within the view (URL-as-state contract broken)** — `frontend/src/views/DiscoverView.vue:75` does not even import `watch`; `route.query` is parsed once at setup (`:88`) and fetched once in `onMounted` (`:339-342`). Every tab/page/filter/search change calls `router.push({ query })` (`commitState` `:111-118`) — new history entries — but there is no `watch(() => route.query, …)`. **Trigger:** page 1→2→3 (or filter, or search) then press Back — the URL reverts but the reused component keeps showing the newer state. README explicitly promises Back restores Discover state (reload works only because the component remounts). **Fix:** `watch(() => route.query, q => { re-parse with parseDiscoverState; refetch })` with an echo guard (skip when `serializeDiscoverState(current)` deep-equals `q`) so `commitState`'s own pushes don't double-fetch. *(Corroborates audit-2.)*

## Medium

- [ ] **M1. Calendar includes already-processed watchlist movies (filter checks a status value that is never set)** — `backend/src/app/modules/calendar/router.py:41` filters `item.status != "available"`, but `Watchlist.status` is only ever `"pending"` (default, `models.py:50`) or `"added"` (`service.py:154`) — nothing writes `"available"`. A movie sent to Radarr therefore appears twice in the agenda: once from the Radarr calendar (`in_library: True`) and once as a watchlist entry (`in_library: False`), contradicting the README ("watchlist movies *not yet in your library*"). **Fix:** filter `item.status == "pending"` (or `not in ("added", "downloading")`). *(Corroborates audit-2.)*

- [ ] **M2. Calendar 500s if a Sonarr record lacks season/episode numbers** — `backend/src/app/modules/calendar/service.py:38-41`: `f"S{season_number:02d}E{episode_number:02d}"` raises `TypeError` when `record.get("seasonNumber")`/`("episodeNumber")` is `None` (specials/announced episodes). Every other field in the normalizer is null-guarded; these two aren't, and there is no per-record try (`router.py:54-57`), so one bad record kills the entire `/api/calendar` response. **Fix:** guard — `subtitle = f"S{sn:02d}E{en:02d}" if isinstance(sn, int) and isinstance(en, int) else None`, appending ` · {episode_title}` only when truthy; or skip/degrade the single record. *(Corroborates audit-1 M2 / audit-2.)*

- [ ] **M3. `process_batch` → `update_season_monitoring(None)` raises a cryptic `TypeError` for the "all seasons" season-update** — `backend/src/app/modules/watchlist/service.py:134-139` passes `selected_seasons=None` when the row's column is NULL; `sonarr/client.py:187` then evaluates `season.get("seasonNumber") in None` → `TypeError: argument of type 'NoneType' is not iterable`, caught by `process_one`'s blanket except and reported as `{"error": "argument of type 'NoneType' is not iterable"}`. Reachable: `POST /api/watchlist` with `is_season_update=true` and no `selected_seasons`, or `PATCH /{tmdb_id}/seasons` with `null` on a season-update row. The documented "None = all seasons" behavior never happens. **Fix:** in `process_one`, treat `None` for a season-update item as "monitor all non-special seasons" (or normalize before the call); defensively widen `update_season_monitoring`'s param to `list[int] | None`. *(Corroborates audit-2.)*

- [ ] **M4. MediaDetailView doesn't refetch on cross-type navigation with the same id** — `frontend/src/views/MediaDetailView.vue:174` watches only `route.params.id`; `/movie/:id` and `/tv/:id` share the component (mediaType from `route.meta`). Navigating `/tv/123` → `/movie/123` (same id, different namespace — collisions are routine) fires nothing, so the page keeps the show's data under the movie URL. **Fix:** `watch(() => [route.params.id, route.meta.mediaType], …)` (or watch `route.fullPath`). *(Corroborates audit-1 M5.)*

- [ ] **M5. Stale-response races in DiscoverView (and MediaDetailView) fetches** — `frontend/src/views/DiscoverView.vue` `performSearch`/`fetchContent` and `MediaDetailView.vue` `fetchMedia` have no request token/AbortController; `switchTab`/`goToPage`/`handleFilterChange`/`submitSearch`/`clearSearch` fire overlapping requests, so a slower older response resolves last and overwrites `items`/`totalPages` for the wrong tab/page (and its `finally` clears `loading` while a newer request is still in flight). **Fix:** monotonic request token — `const id = ++latest; …await…; if (id !== latest) return` before any state write — or an AbortController cancelled at the start of each fetch. *(Corroborates audit-1 H5 / audit-2 — rated Medium here.)*

- [ ] **M6. `get_setting` leaks the DB session on error and swallows all failures** — `backend/src/app/config.py:45-57`: `db.close()` is on the happy path only (leaks the session if `get_raw_value` raises, e.g. Fernet `InvalidToken`), and `except Exception: pass` silently falls back to `.env` (typically empty for UI-managed keys) with zero log output — so a key rotation manifests as "not configured" everywhere with no diagnostics. **Fix:** `try/finally` (or `with SessionLocal() as db:`); narrow the except (`SQLAlchemyError`, `InvalidToken`) and log a warning naming the key before falling back. *(Corroborates audit-1 M6 / audit-2.)*

- [ ] **M7. Per-request arr/TMDB clients are never closed — defeats the documented persistent-connection design** — `backend/src/app/modules/radarr/router.py:10-12` and `sonarr/router.py:10-12` build a fresh client per request via `Depends`; `watchlist/router.py:107,126,170` and `watchlist/service.py:116,120` build throwaway `TMDBClient`/arr clients per call. `BaseArrClient._get_client` lazily creates a persistent `httpx.AsyncClient` (`arr_base.py:14-20`) and `close()` (`:49-52`) has **zero** production callers (no lifespan shutdown). Each request opens a new pool reclaimed only by GC — full TCP/TLS handshake per request, FD churn, "Unclosed AsyncClient" warnings. **Fix:** module-level (or `app.state`) singletons as discovery/calendar/recommendations already do for TMDB, with `await client.aclose()` on shutdown; have the `Depends` factories return the shared instance. *(Corroborates audit-2.)*

- [ ] **M8. Recommendations cache check runs after the full library fetch (every request, even cache hits)** — `backend/src/app/modules/recommendations/router.py:38-64`: the cache key `sig = frozenset(seeds)` requires the watchlist query (`:38`) **plus** `await radarr.get_all_movies()` (`:42`) and `await sonarr.get_all_series()` (`:49`) — two full library dumps — before the TTL check at `:63`. The 6h cache only short-circuits the (already bounded) TMDB fan-out, not the expensive seed/exclusion compute, and the two arr fetches are serial. **Fix:** TTL-only short-circuit before any I/O (accepting up to 6h staleness), or cache the seed/exclusion separately; `gather` the two arr fetches when recomputing. *(Corroborates audit-1 M8.)*

- [ ] **M9. `_enrich_watchlist_item` swallows every non-404 TMDB error → whole list silently degrades** — `backend/src/app/modules/watchlist/router.py:83` catches bare `except Exception`, so 401 (rotated/invalid key), timeouts, and TMDB 5xx all degrade every row to a placeholder titled `TMDB:<id>`. The client contract is that only 404 means "missing" (`_get_or_none`); here a full outage renders the entire watchlist as garbage with no error signal to user or logs. **Fix:** fall back only on 404; let other `TMDBAPIError`/network errors propagate to an `HTTPException` (or log + set a partial-failure flag on `WatchlistResponse`). *(New angle on the enrichment path.)*

- [ ] **M10. The migration test masks the H3 gap (legacy table already has the unguarded columns)** — `backend/tests/test_watchlist_migration.py:19-34` `_create_legacy_table` pre-creates `selected_seasons` and `is_season_update` in the supposedly pre-migration schema and asserts only `priority`/`tags`. The test validates the migration against a fictional legacy schema and can never detect the omission that caused the real v2.4.1 incident — which is why audit-2 read the migration as clean. **Fix:** define the legacy table as the true original (`id, tmdb_id, media_type, added_at, notes, status` only) and assert every column of `Base.metadata.tables['watchlist'].columns` exists after `_migrate_watchlist_columns()`. *(New.)*

- [ ] **M11. Unauthenticated SSRF + error-text disclosure via `/api/settings/test`** — `backend/src/app/modules/settings/router.py:34-46` GETs `{get_setting("radarr_url")}/api/v3/system/status` (and the sonarr equivalent) and returns the outcome — status code on success, raw `str(e)` on failure (`:44,46,64,65`). `radarr_url`/`sonarr_url` are settable via the unauthenticated `PUT /api/settings` and have **no** URL validation (`schemas.py:10-15` types them as plain `Optional[str]`, no `HttpUrl`). A two-step (PUT an internal target, then POST test) turns the server into an internal-network probe oracle; this is primarily an amplifier of C1. **Fix:** validate URLs with Pydantic `HttpUrl` + a scheme/host allowlist (RFC1918 or explicit), and return generic failure messages (log specifics server-side). *(Corroborates audit-1 H3/L5 — rated Medium here since it is gated behind the C1 write.)*
