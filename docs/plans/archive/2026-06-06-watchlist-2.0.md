# Slice: Watchlist 2.0 — priority, notes, tags, sections (Roadmap Tier 2 #4)

**Date:** 2026-06-06
**Planning scope:** Medium (small DB change + endpoint + frontend grouping/editing UI).
**Roadmap item:** `docs/superpowers/specs/2026-06-06-roadmap-design.md` §Tier 2 #4.
**Builds on:** #2 (watchlist sort & filter, v2.8.0) — extends the same pure `watchlistState.js` util and thin-glue `WatchlistView.vue`.

## Goal

Turn the flat watchlist into something organizable: an editable **priority**, the already-existing **notes** field surfaced and editable, optional **tags**, and **section grouping by priority**.

**Acceptance (from roadmap):** Set a priority and a note on an item; both persist; the list groups by priority.

## Contract (frozen here so Level-1 tasks can proceed in parallel)

- **`priority`** — integer, three levels: `1` = High, `0` = Normal (**default**), `-1` = Low. Migration default `0` ⇒ existing items land in **Normal**. Section order is descending priority value: **High → Normal → Low**.
- **`tags`** — `list[str]`, stored as a JSON-encoded string in a nullable `TEXT` column (same pattern as `selected_seasons`). Empty/absent ⇒ stored `NULL`, surfaced as `[]`.
- **`notes`** — already exists on the model/schema; this slice makes it editable.
- **PATCH body** (`/api/watchlist/{item_id}`): `{ priority?, notes?, tags? }`, **partial** — only keys present in the request body are written (`model_dump(exclude_unset=True)`). `priority` constrained to `Literal[-1, 0, 1]`. Keyed by primary-key `id` (unambiguous; `tmdb_id` can collide across movie/show rows).
- **`WatchlistItem` response** gains `priority: int` and `tags: list[str]` (and keeps existing `notes`).
- **URL state:** new `groupBy ∈ {'none','priority'}` (default `'none'`), mirrored to query key `group`; defaults omitted (consistent with existing `serializeWatchlistState`).

## Critical design note — the migration is mandatory (do not skip)

`init_db()` calls only `Base.metadata.create_all`, which **creates missing tables but never adds columns to an existing table**. Adding `priority`/`tags` to the model therefore breaks the live NAS SQLite DB on read — the exact "no such column → 500 on watchlist load" incident that shipped as the v2.4.1 hotfix when `is_season_update` was added. The roadmap's phrase "same lightweight migration approach as the prior `is_season_update` column add" is misleading: that add had **no** programmatic migration (verified — no `ALTER`/`PRAGMA`/migration code exists anywhere in `backend/src/app`). This slice does it correctly with an idempotent guarded `ALTER`.

## T1 — Backend: model columns + idempotent migration + response schema

- `backend/tests/test_watchlist_migration.py` — NEW.

**Change**
- `models.py` → add to `Watchlist` (after `is_season_update`):
  - `priority: Mapped[int] = mapped_column(Integer, default=0)`  (1=High, 0=Normal, -1=Low)
  - `tags: Mapped[str | None] = mapped_column(Text, nullable=True)`  (JSON array of strings, or null)
- `database.py` → add an idempotent migration and call it from `init_db()` **after** `create_all`:
  ```python
  from sqlalchemy import create_engine, inspect, text
  ...
  def _migrate_watchlist_columns() -> None:
      """Add columns introduced after a table already exists (SQLite ALTER ADD COLUMN). Idempotent."""
      inspector = inspect(engine)
      if "watchlist" not in inspector.get_table_names():
          return
      existing = {c["name"] for c in inspector.get_columns("watchlist")}
      stmts = []
      if "priority" not in existing:
          stmts.append("ALTER TABLE watchlist ADD COLUMN priority INTEGER NOT NULL DEFAULT 0")
      if "tags" not in existing:
          stmts.append("ALTER TABLE watchlist ADD COLUMN tags TEXT")
      if stmts:
          with engine.begin() as conn:
              for s in stmts:
                  conn.execute(text(s))

  def init_db():
      """Create all tables, then apply lightweight additive migrations."""
      Base.metadata.create_all(bind=engine)
      _migrate_watchlist_columns()
  ```
  (Fresh DBs: `create_all` already builds the full table ⇒ migration is a no-op. Existing DBs: migration adds only the missing columns. AUTHORIZED schema migration.)
- `schemas.py` → in `WatchlistItem` add: `priority: int = 0` and `tags: list[str] = []`. (`notes` already present. Do NOT touch `WatchlistAdd` — new items default; details are set later via PATCH.)

**Tests (TDD — write first, watch fail, then implement)**
- `test_migration_adds_columns_to_legacy_table`: build an engine, create a `watchlist` table WITHOUT `priority`/`tags` via raw `CREATE TABLE` (id, tmdb_id, media_type, added_at, notes, status, selected_seasons, is_season_update), insert one legacy row, then point the migration at it and assert both columns now exist (`inspect(...).get_columns`) and a `SELECT priority, tags FROM watchlist` returns `(0, None)` for the legacy row.
- `test_migration_is_idempotent`: running the migration twice does not raise and leaves exactly one `priority`/`tags` column.
- `test_fresh_create_all_has_new_columns`: `create_all` on a clean engine yields a `watchlist` table containing `priority` and `tags`.
- Use the in-memory-engine pattern from `tests/test_watchlist_service.py`. To exercise the helper against an arbitrary engine, factor it to accept an engine arg internally or monkeypatch `app.database.engine` in the test (state which in impl).
- **Verify-at-impl:** confirm SQLAlchemy `inspect`/`text` import path under the installed 2.x (`Mapped`/`mapped_column` already in use ⇒ 2.x) and that `ALTER TABLE … ADD COLUMN … NOT NULL DEFAULT 0` is accepted by the bundled SQLite.

**Change**
- `service.py` → add:
  ```python
  def update_details(self, item_id: int, fields: dict) -> Watchlist | None:
      """Partial update of priority / notes / tags by primary-key id. `fields` holds only provided keys."""
      item = self.get_by_id(item_id)
      if not item:
          return None
      if "priority" in fields:
          item.priority = fields["priority"]
      if "notes" in fields:
          item.notes = fields["notes"]
      if "tags" in fields:
          tags = fields["tags"]
          item.tags = json.dumps(tags) if tags else None   # [] or None -> NULL
      self.db.commit()
      self.db.refresh(item)
      return item
  ```
- `router.py`:
  - Add a small `_parse_tags(raw: str | None) -> list[str]` mirroring `_parse_seasons` (returns `[]` on null/garbage).
  - In **both** branches of `_enrich_watchlist_item` (success + `except`) pass `priority=item.priority` and `tags=_parse_tags(item.tags)`.
  - Add an inline request model next to `UpdateSeasonsRequest`:
    ```python
    class UpdateDetailsRequest(BaseModel):
        priority: Literal[-1, 0, 1] | None = None
        notes: str | None = None
        tags: list[str] | None = None
    ```
    (import `Literal` from `typing`.)
  - Add the endpoint **after** the `/{tmdb_id}/seasons` PATCH and **before** the `DELETE /{item_id}` (unambiguous anyway — different method; keep grouped). Use `exclude_unset` so absent keys are untouched:
    ```python
    @router.patch("/{item_id}/details")
    async def update_watchlist_details(item_id: int, data: UpdateDetailsRequest,
                                       service: WatchlistService = Depends(get_service)):
        fields = data.model_dump(exclude_unset=True)
        item = service.update_details(item_id, fields)
        if not item:
            raise HTTPException(status_code=404, detail="Item not found")
        tmdb = TMDBClient(api_key=settings.tmdb_api_key)
        return await _enrich_watchlist_item(item, tmdb)
    ```
    (Path `/{item_id}/details` avoids any overlap with the bare `/{item_id}` DELETE and keeps intent explicit.)

## Risks & mitigations

- **Live-DB breakage (highest risk):** mitigated by the idempotent additive migration in `T1` + a legacy-table migration test. Without it, the watchlist page 500s on the NAS exactly like v2.4.1.
- **Route ambiguity:** `/{item_id}/details` (PATCH) is distinct from `/{tmdb_id}/seasons` (PATCH) and the bare `/{item_id}` (DELETE) — no overlap.
- **Partial-update semantics:** `exclude_unset` ensures editing one field never wipes the others; covered by a router test.
- **Default priority for existing rows:** `ALTER … DEFAULT 0` ⇒ all current items become "Normal", matching user expectation.
